# Syndrome Decoding as Validator Consensus: a work-in-progress report from the Antigence subnet

**Project:** Antigence Subnet
**Scope:** three linked milestones shipped 2026-04-21 → 2026-04-24
**Audience:** Bittensor subnet builders, validator developers, technically-curious readers
**Repository:** https://github.com/biobitworks/antigence-subnet

## What this is

A short write-up of three linked milestones that built a replay-verifiable scoring sidecar for a Bittensor validator, then promoted it into production behind an opt-in flag.

**Two-sentence version.** We adapted error-correction-code "syndrome decoding" as a framing for anomaly classification in a scoring validator — each miner output becomes a codeword, deviations become syndromes, a versioned lookup maps syndromes to named anomaly classes. We also built a local SHA-256-linked audit log, three named failure-pattern detectors, and a replay harness; the whole stack promoted to production behind an opt-in flag with a one-command revert.

**What this isn't.** Not yet tested on a real testnet against real miners. Not claiming this improves incentive design — only that it gives users and third-party auditors a verifiable answer to "what did the validator do, and can I check?"

## The problem this addresses

Bittensor validator consensus is imperfect by construction. Independent validators process the same miner responses and should agree on scoring, but in practice there's drift: float-order accumulation, non-deterministic RNG seeding, and silent type coercions all cause small disagreements that compound. Yuma Consensus tolerates this statistically, but it's a tax on the signal. Secondarily, once scores are set, there's no local audit — you have the on-chain `set_weights` result, and that's it. A disputed round ("why was miner X scored so low?") has to be reconstructed from scratch.

## What was built

Three milestones:

**Deterministic scoring layer.** A standalone Python package that can be tested in isolation with no Bittensor SDK installed. Frozen dataclasses for all score state; canonical JSON serialization with NaN/Inf rejection; SHA-256 chain linking rounds in a local JSONL file; deterministic replay harness that reproduces frozen state byte-for-byte from a recorded log; tamper detection that rejects record mutation, reordering, or deletion. 74 fixture-based tests pass under a minimal venv.

**Integration against a verbatim copy of production** (experiment-first, no production edits). A bridge wraps the existing `get_rewards()` numpy output and emits frozen round state to the chain. A convergence hook runs three detectors — oscillation (gaming), metastability (mediocre plateau), convergence-failure (cross-validator divergence) — over the chain and emits structured JSON events plus Prometheus counters. A config layer makes the whole pipeline opt-in via `validator.audit.enabled=false` default, with CLI and TOML precedence matching existing config patterns. Production code remained byte-identical throughout this milestone: a full `git diff` against the prior tag returned 0 lines.

**Promotion to production.** Single atomic commit moves the four new modules into `antigence_subnet/validator/` and replays three surgical diffs onto production `forward.py`, `base/validator.py`, `validate_config.py`. Full pytest suite under real Bittensor SDK: 1,730 pass, 0 fail. A revert script reverses the whole migration in one command; dry-run and full-cycle tested. The audit chain remains `enabled=false` by default — no behavior change for existing validators unless they opt in.

## What's distinctive (calibrated against a 16-subnet survey)

A survey of 16 Bittensor subnets grounded the novelty claims against real exemplars. Calibrated findings:

- **Syndrome-based anomaly classification.** Miner outputs encoded as fixed-length codewords; a pure, deterministic function computes a ternary bucket signature; a versioned JSON lookup table maps signatures to named anomaly classes, with unknowns mapping to `"unclassified"` (never raising). No surveyed subnet uses error-correction-code vocabulary this way. This pattern is imported from coding theory, not invented here — credit below.

- **SDK-free scoring package.** The deterministic-scoring module has zero `import bittensor` anywhere. It tests under bare Python + numpy. Most subnets' validator logic is inseparable from the SDK; ours can be verified in isolation by someone who hasn't installed the chain stack. A closely-related pattern: SN68 Nova's deterministic PSICHIC oracle achieves determinism by running the same oracle on the same molecule, by construction. We reach determinism from a different angle — architectural separation rather than oracle equivalence.

- **Local tamper-evident audit chain.** A local, user-owned SHA-256-linked audit parallel to the on-chain submission. SN17 404-GEN has a public git-backed audit trail addressing a similar threat model — different form (public vs local), same goal. We're not claiming first. SN4 Targon's "verifiable inference" may be another parallel — we could not pull enough implementation detail from public docs to verify. Anyone considering this a novelty claim should check Targon first.

- **Three named failure-pattern detectors.** Oscillation, metastability, convergence-failure — specific named modes rather than a generic anomaly score. The closest existing thing in surveyed subnets is Opentensor's own weight-copier detector, which operates at the consensus level rather than per-miner trajectory. Soft claim: this decomposition into named modes is uncommon in surveyed subnets, not unheard of.

Specifically **not novel, intentionally**: SHA-256 hash chains (Git, blockchain, banking), canonical JSON (every crypto protocol), frozen dataclasses (standard Python), opt-in config toggles, Prometheus metrics, structured logs, deterministic replay (banks and databases have done this for decades). Our contribution is applying these standard primitives to a Bittensor subnet's scoring problem coherently, under an error-correction-code framing.

## Development discipline worth naming

**Experiment-first promotion with a one-command revert.** The integration was built on verbatim copies of production. Promotion to production was a single atomic commit with an accompanying revert script that undoes the migration completely. A written risk-acceptance record was filed before promotion; reviewed and signed off before the migration ran.

**Red-team review writes the acceptance criteria.** For commits that modify production, a separate red-team review pass produces the plan with concrete acceptance tests — each anticipated failure mode becomes a specific test. The promotion plan had 12 halt conditions; if any fired, the migration would revert and not tag.

## Evidence on the ground

- **1,730 tests pass under real Bittensor SDK** (full pytest suite). 38 specific to the integration; the rest baseline from prior milestones.
- **212 tests pass under a minimal Python venv** without the Bittensor SDK installed — this is the portability demonstration for the SDK-free claim. Breakdown: 174 tests in the deterministic-scoring package itself (standalone), 38 tests for the integration layer.
- **Production code was byte-identical to the pre-integration tag throughout the experiment milestone.** The subsequent promotion is the one intentional change; `git diff` against a pre-integration tag at that point returns exactly the migration diff.
- **Revert script dry-run** exits clean and reports the migration commit SHA. Full-cycle ephemeral-worktree test returned the tree to a 0-line diff against the pre-promotion tag.

## What's not verified yet

- **No real-testnet run.** The scoring layer and detectors have never touched a funded Bittensor wallet. A testnet dry-run is scoped with a runbook and a scripted adversarial fixture miner, but not executed. Running it requires a funded testnet wallet and a registered netuid.
- **Convergence-hook latency is 57–65 ms median** on a 20-round × 256-miner synthetic fixture. Aspiration was 5 ms. Production hook is non-blocking (try/except wrap), rounds happen per-tempo (minutes), so operationally fine — but anyone running at tight tempo should consider the three documented optimization paths (incremental in-memory trajectory buffer, faster JSON decoder, narrower default window).
- **Cross-validator convergence-failure detection** currently runs over a *simulated* second replica (two trajectories in one test fixture). Real multi-validator deployment would need actual cross-validator score ingestion.
- **Stub-backed tests exist.** 38 of 212 minimal-venv tests use bittensor stubs. They prove module structure, not runtime behavior against the real SDK. The full-suite 1,730-pass run under real SDK is the proof-of-life for production.

## How to verify

The standalone deterministic-scoring package installs under bare Python + numpy + pytest and runs its own test suite without the Bittensor SDK. Expected: 174 tests pass. Any recorded audit-chain `chain.jsonl` can be fed to the package's `verify_chain` function for a yes/no integrity answer; tamper with any record and the call raises at the compromised record.

```bash
git clone https://github.com/biobitworks/antigence-subnet.git
cd antigence-subnet
python3.13 -m venv .venv
.venv/bin/pip install fastapi uvicorn pydantic aiohttp pytest numpy
.venv/bin/python -m pytest tests/validator/deterministic_scoring/ -v \
  --confcutdir=tests/validator/deterministic_scoring \
  --rootdir=tests/validator/deterministic_scoring
```
